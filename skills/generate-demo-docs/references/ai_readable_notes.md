# 如何写适合 AI 阅读的深层说明 / AI-Readable Technical Notes Guide

> 章节 7（复杂脚本深度解读）的写作指南。
> 目标：让另一个 AI 读完这份文档后，能直接基于它分析、修改、或重构代码，
> 不需要再读源码就能做出正确判断。

---

## 核心原则

### 原则 1：用语义，不用引用

❌ 不好：
> "详见第 42 行的 `_synthesize_audio_from_lines` 函数"

✅ 好：
> "`_synthesize_audio_from_lines` 函数（位于 `embedded_server_main.py`）接收一个对话行列表，为每行分别调用 TTS 引擎生成音频片段，再用 ffmpeg 拼接为完整音频文件。"

AI 读文档时不能"跳转到第 42 行"，必须在文本中完整表达意图。

---

### 原则 2：显式说明"为什么"，不只说"是什么"

❌ 不好：
> "这里使用 OrderedDict 作为缓存"

✅ 好：
> "这里使用 `OrderedDict` 而不是普通 `dict` 作为缓存，是因为 Python 3.7+ 的普通 dict 虽然有序，但没有内置的 LRU（最近最少使用）淘汰接口。`OrderedDict.move_to_end()` 方法让我们可以将最近访问的条目移到末尾，从头部弹出最旧的条目，实现 O(1) 的 LRU 操作。"

---

### 原则 3：说明隐式约定（最容易被忽略）

隐式约定是"代码里没有注释，但你不遵守就会出错"的规则。

示例：
- "此函数假设输入列表至少有 1 个元素，传入空列表会触发 IndexError"
- "文件路径必须是绝对路径，相对路径不会报错但结果不可预测"
- "此配置在模块加载时只读取一次，运行时修改不生效"
- "输出目录必须在调用前存在，函数不会自动创建"

---

### 原则 4：标注重要的边界条件

AI 分析代码时最需要知道什么情况下代码会"失败"或"表现异常"。

格式：
```
⚠️ 边界条件：
- 输入为空时：{行为}
- 文件不存在时：{行为：报错/返回空/创建}
- 超时时：{行为}
- 并发场景：{线程安全/非线程安全}
```

---

### 原则 5：函数签名必须完整描述

对于关键函数，不能只写"做 X"，要完整说明：

模板：
```
函数名：`function_name(param1, param2, *, keyword_arg=default)`
  - param1 (str)：输入文本，必须是 UTF-8 编码，长度不超过 1000 字符
  - param2 (list[str])：候选列表，至少包含 1 个元素，允许空字符串
  - keyword_arg (int, 默认 5)：并发数量上限，范围 1-20
  - 返回值：(str, list) 元组，第一个是处理后的文本，第二个是匹配项列表
  - 可能的异常：ValueError（param1 为空）、TimeoutError（超过 30 秒）
```

---

### 原则 6：用数据格式图说明复杂结构

当函数传递的数据是复杂结构时，用示例说明：

❌ 不好：
> "返回一个包含对话信息的字典"

✅ 好：
```python
# 返回值结构示例：
{
    "status": "success",           # str，"success" | "error" | "partial"
    "lines": [                     # list，每个对话行
        {
            "speaker_id": 1,       # int，说话人编号（从 1 开始）
            "text": "你好，...",   # str，对话文本
            "audio_path": "/tmp/...", # str | None，生成的音频文件路径
            "duration_sec": 2.3,   # float | None，音频时长（秒）
        }
    ],
    "error": None                  # str | None，错误信息（status="error" 时有值）
}
```

---

### 原则 7：区分"设计决策"和"历史遗留"

AI 在分析代码时需要判断某段代码是"有意为之"还是"历史遗留问题"。

格式：
- `[设计决策]`：这样写是有意为之，有特定原因
- `[历史遗留]`：这段代码可能是历史遗留，可能已经不是最优解
- `[技术债]`：这里已知有问题，但暂时没有修复
- `[待优化]`：功能正确，但性能或可读性有改进空间

示例：
> `[技术债]` 这里使用了 `time.sleep(0.1)` 轮询等待，应该用事件通知替代，但改动较大暂未处理。

> `[设计决策]` 使用 ffmpeg 子进程而不是 pydub 在内存中合并音频，是因为大量音频片段会导致内存峰值过高。

## AI 可读的优缺点写法

### 不好的写法（模糊）
> "代码质量一般，有待提升"

### 好的写法（具体，AI 可以直接基于此判断）

```
#### `embedded_server_main.py` 代码质量评估

**实际存在的问题**（可以直接基于这些信息进行重构）：

1. **全局状态过多**：`_manifest_cache`、`_ffmpeg`、`_extract_done` 等 6 个全局变量散布在文件顶部，
   没有封装成类，导致测试困难。如果要写单元测试，无法轻松重置这些状态。
   → 建议：将全局状态封装到 `BundleServer` 类内。

2. **错误处理不统一**：部分函数返回 None 表示失败，部分抛出异常，部分返回错误字符串。
   调用方必须知道每个函数的错误约定，否则会漏掉错误。
   → 建议：统一使用异常，或统一使用 Result 类型。

3. **函数过长**：`_synthesize_audio_from_lines` 函数有 150+ 行，包含了太多逻辑。
   → 建议：拆分为 `_generate_segments()` + `_concatenate_audio()` + `_cleanup_temp()` 三个独立函数。
```

---

## 深度解读的推荐长度

| 脚本复杂度 | 推荐字数 |
|-----------|---------|
| 高复杂度（> 300 行） | 800-1500 字 |
| 中等复杂度（100-300 行） | 400-800 字 |
| 低复杂度（< 100 行） | 200-400 字 |

超过 1500 字时，要检查是否有重复信息，进行精简。

## 维护者建议的写法

章节 7 末尾的"维护者建议"要足够具体，让 AI 可以直接执行：

❌ 太模糊：
> "建议优化代码结构"

✅ 可执行：
> "如果需要修改音频合并逻辑，优先修改 `_synthesize_audio_from_lines` 函数（第 1250 行附近）。
> 修改前必须了解 ffmpeg concat demuxer 格式（临时文件 `.concat_list.txt` 的格式规范）。
> 修改后，用超过 10 行对话的场景验证：音频完整性、临时文件是否清理、内存峰值是否合理。"
