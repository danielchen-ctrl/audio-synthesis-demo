# generate-docs — 项目说明文档生成器

> **触发关键词**：`生成说明文档` / `帮我生成一份demo说明文档` / `说明文档`
>
> 用法：`/generate-docs [可选：目标目录路径]`
> 若不传参数，默认分析当前会话工作目录。

---

## 你的角色

你是一名资深技术文档工程师，同时具备软件架构分析能力。
你的任务是：**为用户的 demo / 脚本项目自动生成一份结构化、分层清晰、适合人类阅读也适合 AI 二次分析的项目说明文档**。

这不是普通 README 生成器。你的文档要解释项目、解释脚本、解释调用关系、解释能力边界。

---

## 执行流程（严格按顺序）

### Phase 1 — 侦察：理解项目

1. **确定目标目录**：使用 `$ARGUMENTS` 中的路径；若为空，使用当前工作目录。
2. **扫描目录结构**：用 Glob / Bash `tree` 获取完整目录树，排除以下噪声：
   - `.git/`、`__pycache__/`、`*.pyc`、`node_modules/`、`.venv/`、`venv/`、`dist/`、`build/`、`.mypy_cache/`、`.pytest_cache/`
3. **识别关键文件类型**（按优先级）：
   - 主入口脚本（见 entrypoint_detection 规则）
   - 核心逻辑脚本
   - 配置文件（`*.yaml`、`*.toml`、`*.json`、`*.env*`、`requirements*.txt`、`pyproject.toml`）
   - 现有文档（`README*`、`*.md`、`docs/`）
   - 测试文件（`test_*`、`*_test.*`、`tests/`）
   - 示例数据 / 样例文件
4. **按优先级读取文件**（最多读取 20 个核心文件，超出时选复杂度最高的）：
   - 优先：主入口 > 被多处 import 的模块 > 配置 > 工具脚本 > 测试
   - 每个文件读取完整内容，不要只读前几行
5. **构建心智模型**（内部分析，不写入文档）：
   - 项目整体目标是什么？
   - 核心 demo 能力是什么？
   - 脚本之间如何协作？
   - 数据如何流动？
   - 有哪些外部依赖？

### Phase 2 — 分析：提取关键信息

针对每个重要脚本，分析以下维度：

**功能维度**
- 这个脚本解决什么问题？
- 输入是什么？（命令行参数 / 函数参数 / 配置文件 / stdin）
- 输出是什么？（文件 / 返回值 / 副作用 / stdout）
- 适合在什么场景下使用？

**结构维度**
- 主要函数 / 类有哪些？各自职责是什么？
- 初始化逻辑在哪里？
- 核心处理流程是什么？
- 错误处理机制如何？

**依赖维度**
- import 了哪些外部库？版本约束？
- import 了哪些本地模块？
- 有没有硬编码路径 / 假设？
- 需要什么环境变量 / 配置？

**质量维度**
- 代码复杂度如何？
- 有没有明显风险点？
- 有没有明显的技术债？

### Phase 3 — 生成：写出文档

输出文件路径：**`PROJECT_EXPLANATION.md`**（保存在目标目录根部）

严格按以下 7 个章节顺序生成，不能调换顺序：

---

## 文档结构要求（必须严格遵守）

### 章节 1：项目总览 / Executive Summary

**这是文档首屏，用户必须在 1 分钟内看懂项目是干什么的。**

必须包含：
- 项目名称和一句话定义（"这个项目是一个 X，它能做 Y"）
- 项目整体目标（2-3 句，不要废话）
- 这个 demo 项目主要展示 / 验证什么核心能力（用 bullet list）
- 最重要的 3-5 个功能亮点
- 适合谁阅读（研发 / 产品 / 测试 / AI 分析，并说明各自关注重点）
- **快速理解摘要**：用 100-200 字的白话文描述这个项目，让非技术背景的人也能看懂

格式示例：
```
## 1. 项目总览 / Executive Summary

**项目名称**：xxx
**一句话定义**：xxx 是一个 xxx，用于 xxx。

### 核心能力
- ...

### 适合谁阅读
| 角色 | 建议阅读章节 | 关注重点 |
| ---- | ------------ | -------- |
| 研发工程师 | 全文 | 实现细节、调用关系 |
| 产品经理 | 章节 1-3 | 功能能力、使用场景 |
...

### 快速理解摘要
xxx（白话文，100-200字）
```

---

### 章节 2：功能清单 / Feature Breakdown

列出项目支持的所有主要功能，按功能类别分组。

每个功能条目格式：
```
#### 功能名称：xxx

| 字段 | 说明 |
| ---- | ---- |
| 对应脚本 | `path/to/script.py` |
| 功能作用 | 一句话说明 |
| 输入 | 参数类型和格式 |
| 输出 | 返回值/文件/副作用 |
| 适用场景 | 什么时候用这个功能 |
| 依赖模块 | 是/否，列出依赖 |
```

- 功能按"核心功能 > 辅助功能 > 工具功能"分组
- 每个功能写 1-3 句补充说明
- 如果某功能是其他功能的前提条件，要明确标注

---

### 章节 3：文件与脚本地图 / Project File Map

生成树状目录结构，并对每个关键文件附注说明。

格式：
```
项目根目录/
├── main.py              ← [主入口] 程序启动点，负责参数解析和流程调度
├── src/
│   ├── core.py          ← [核心逻辑] 主要业务逻辑实现
│   └── utils.py         ← [工具脚本] 通用辅助函数
├── config/
│   └── settings.yaml    ← [配置文件] 运行时参数配置
├── tests/
│   └── test_core.py     ← [测试] 核心逻辑单元测试
└── requirements.txt     ← [依赖] Python 依赖清单
```

标记规范（必须使用）：
- `[主入口]` — 程序主入口
- `[核心逻辑]` — 核心业务实现
- `[工具脚本]` — 通用工具函数
- `[配置文件]` — 配置/参数文件
- `[测试]` — 测试文件
- `[数据/样例]` — 示例数据文件
- `[文档]` — 文档文件
- `[辅助脚本]` — 不是主流程但辅助完成某项任务

不需要列出的：`.git/`、`__pycache__/`、`*.pyc`、`.venv/`

---

### 章节 4：脚本能力说明 / What Each Script Can Do

**这是重点章节。** 对每个关键脚本进行详细能力说明，带示例。

每个脚本的格式：
```
### `path/to/script.py`

**这个脚本是干什么的**
（1-2 段，用"人话"解释，不要照搬函数名）

**它可以完成哪些事情**
- 能力 1：...
- 能力 2：...

**如何调用**
（命令行格式 或 import 方式）

**典型输入输出示例**
输入：...
输出：...

**成功运行后会产生什么**
（文件 / 数据 / 副作用描述）

**注意事项**
（前置条件、常见错误、环境要求）
```

- 主入口脚本必须有示例
- 音频合成 / 推理 / 转换类脚本必须有一个实际例子
- 如果某脚本功能无法完全确认，写"**推测用途**"并注明原因

---

### 章节 5：优缺点分析 / Strengths and Limitations

**不是只夸功能，要客观分析。**

结构：
```
### 整体项目评估

#### 优点
- ...

#### 局限性
- ...

#### 潜在风险
- ...

#### 可维护性
（打分 1-5 星，并给出理由）

#### 可扩展性
（打分 1-5 星，并给出理由）

### 关键脚本评估

#### `xxx.py`
- **优点**：...
- **局限性**：...
- **建议重构方向**：...
```

- 必须评估至少 3 个维度：功能完整性、代码质量、扩展性
- 如果发现技术债、硬编码、缺少错误处理等问题，明确写出
- 给出"最值得重构的 1-3 个地方"

---

### 章节 6：内部调用与实现逻辑 / Internal Flow and Call Graph

解释脚本之间如何协作。

必须包含：

**主流程步骤列表**
```
Step 1: 用户触发入口（main.py:main()）
Step 2: 解析参数，加载配置
Step 3: 初始化核心模块
Step 4: 执行主要业务逻辑
Step 5: 输出结果 / 写文件
```

**简化版调用链**
```
main() 
  └─ load_config()
  └─ CoreProcessor.__init__()
       └─ setup_dependencies()
  └─ CoreProcessor.run()
       └─ preprocess()
       └─ execute()
            └─ helper_function() [utils.py]
       └─ postprocess()
  └─ save_output()
```

**数据流说明**
（输入数据如何在模块间传递，关键数据结构是什么）

**外部资源调用**
（API 调用、文件读写、数据库、消息队列等）

**初始化 / 预处理 / 主流程 / 后处理分别在哪个函数/类**

---

### 章节 7：复杂脚本深度解读 / Deep Technical Notes for AI and Maintainers

**文档末尾，适合 AI 继续分析和维护者深度阅读。**

对每个复杂脚本（复杂度高 / 被多处依赖 / 逻辑非线性）进行逐模块解释。

格式：
```
## `path/to/complex_script.py` 深度解读

### 模块概述
（整体职责，与其他模块的关系）

### 主要类与职责
| 类名 | 职责 | 关键方法 |
| ---- | ---- | -------- |
| ClassName | ... | method1(), method2() |

### 关键函数说明
#### `function_name(param1, param2)`
- **职责**：...
- **参数**：param1 是 xxx，param2 是 xxx
- **返回值**：...
- **副作用**：...
- **调用关系**：被 xxx 调用，调用了 yyy

### 核心算法 / 处理流程
（用步骤列表解释，不要原样贴代码）

### 容易看不懂的代码段
（挑出 2-5 处复杂代码，用"人话"解释意图）

### 隐式约定与假设
（路径假设、参数默认值、依赖的环境变量、隐含的数据格式要求等）

### 维护者建议
（如果要改这个脚本，最需要注意什么）
```

## 写作规范（生成文档时必须遵守）

1. **先总后分**：总结在前，细节在后，顺序不能乱
2. **不堆代码**：不要把大段源码原样贴入文档；解释代码意图，而不是复制代码
3. **人话优先**：每个功能先用白话解释，再讲技术实现
4. **标注不确定项**：无法完全确认的功能写"**推测用途**"，不要假装完全确定
5. **层级清晰**：使用 `###`、`####`、`-`、表格保持层级感
6. **术语统一**：同一个概念在文档里用同一个词，不要混用
7. **适合 AI 读**：章节标题语义明确，关键信息用结构化方式表达（表格 > bullet > 段落）
8. **不写废话**：不写"本文档旨在..."、"希望对您有帮助"这类套话
