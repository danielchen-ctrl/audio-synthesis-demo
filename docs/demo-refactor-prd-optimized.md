# 重构 demo 功能 PRD（优化版，面向 Codex 直接执行）

## 1. 文档目标

本文档用于指导对现有 `场景对话 Demo` 进行重构。

目标不是把它做成完整平台，而是在保留当前 Demo：

- 可演示
- 可本地运行
- 可局域网分享
- 保持单页结构

的前提下，把现有链路：

```text
生成场景对话文本 -> 手动编辑 -> 生成 mp3 音频 -> 下载文本 / 下载音频
```

重构为一个更清晰、可扩展、可维护的 Demo。

本文档已经修复上一版 PRD 中影响实施的关键问题：

1. 统一多人对话文本格式
2. 明确音频生成前的文本最大长度硬阈值
3. 明确下载文本的唯一权威实现
4. 明确重置时是否恢复人物画像默认值
5. 将“新接口命名”从硬要求降级为可选优化
6. 不再依赖聊天上下文中的截图，补充为可直接执行的文字布局约束

## 2. 产品定位

这是一个本地可运行、可局域网分享、面向演示的场景对话生成 Demo。

它不做以下内容：

- 登录注册
- 多用户协作
- 历史工作台
- 完整任务中心
- 云端存储
- 权限系统
- 完整素材管理系统
- Speaker 音色配置系统
- 上传文件转音频

它只解决一个核心问题：

> 用户在一个页面中，配置人物画像与对话设置，生成一段可编辑的多人对话文本，再把当前文本合成为 mp3 音频，并下载文本与音频结果。

## 3. 本次重构目标

### 3.1 必须达成

1. 保留现有单页 Demo 形态
2. 保留局域网分享能力

7. 明确“当前编辑中的文本”是音频生成和文本下载的唯一依据
8. 明确“文本修改后旧音频失效”
9. 页面要清楚展示文本状态、音频状态、当前文本文件、当前音频文件

### 3.2 可选优化

1. 更清晰的空状态和提示语
2. 更清晰的按钮置灰逻辑
3. 统一返回结构中的辅助字段

### 3.3 暂不做

1. 不改成多页面应用
2. 不引入数据库
3. 不增加复杂任务队列
4. 不增加上传音频能力

## 4. 布局与兼容性约束

本文档不依赖聊天中的截图附件，以下文字约束即为权威视觉要求。

### 4.1 页面整体结构

页面仍为单页，自上而下按以下区块排列：

1. 页头区
2. 人物画像区
3. 对话设置区
4. 操作区
5. 运行状态区
6. 生成结果区

### 4.2 对话设置区必须保留的字段与顺序

字段顺序必须为：

1. 场景说明
2. 对话核心内容
3. 人物数量（含 Owner）
4. 字数限制

要求：

- 字段名称保持不变或只做极小调整
- 视觉上仍属于同一个卡片区块
- 仍然是文本生成前的核心输入区

### 4.3 操作按钮区必须保留的按钮与顺序

按钮顺序必须为：

1. 生成场景对话文本
2. 生成 mp3 音频
3. 下载文本
4. 下载音频
5. 重置

要求：

- `生成场景对话文本` 为主按钮
- `生成 mp3 音频` 在无文本时必须不可用
- `下载文本` 在无可下载文本时不可用
- `下载音频` 在无有效音频时不可用
- `重置` 始终可点击，但点击后必须二次确认

## 5. 核心用户流程

```text
填写人物画像
-> 填写对话设置
-> 点击“生成场景对话文本”
-> 系统生成文本
-> 用户可在结果区手动编辑
-> 点击“生成 mp3 音频”
-> 系统基于当前编辑后的文本生成音频
-> 用户下载文本 / 下载音频
```

核心规则：

- 文本生成与音频生成是两个明确步骤
- 文本生成成功后，用户可以只下载文本，不强制生成音频
- 生成音频时必须以当前编辑区文本为准
- 文本一旦变化，旧音频立即失效

## 6. 人物画像配置

### 6.1 字段

1. 职业
2. 工作内容
3. 职级
4. 使用场景

### 6.2 要求

1. 保持当前下拉选择形式
2. 保持默认值
3. 所选内容参与文本生成请求
4. 不要求在结果区显式展示这些字段

## 7. 对话设置

### 7.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| 场景说明 | 多行输入 | 是 | 描述具体对话发生的场景 |
| 对话核心内容 | 多行输入 | 是 | 描述对话必须覆盖的重点 |
| 人物数量（含 Owner） | 选择器 | 是 | 参与对话的人数，包含 Owner |
| 字数限制 | 输入框 | 是 | 目标文本长度上限，单位为字 |

### 7.2 输入规则

1. 场景说明不能为空
2. 对话核心内容不能为空
3. 人物数量必须大于等于 2
4. 字数限制必须为正整数
5. 推荐默认值：字数限制 `1000`
6. 推荐默认值：人物数量 `3`

说明：

- 人物数量表示生成对话时的人物数，不是音轨数
- 本次 Demo 不引入 Speaker 配置界面

## 8. 文本生成

### 8.1 按钮名称

`生成场景对话文本`

### 8.2 生成前校验

必须校验：

1. 职业已选择
2. 工作内容已选择
3. 职级已选择
4. 使用场景已选择
5. 场景说明非空
6. 对话核心内容非空
7. 人物数量有效
8. 字数限制有效

### 8.3 多人对话文本格式

本次重构统一采用以下**唯一权威格式**：

```text
Speaker 1: ...
Speaker 2: ...
Speaker 3: ...
```

规则：

- 必须使用 `Speaker N:` 前缀
- `N` 从 `1` 开始，连续递增
- 不能使用 `Owner:`、`角色 2:`、`A:`、`B:` 这类替代格式作为最终持久化格式

原因：

- 当前后端文本保存与音频生成链路按 `Speaker N:` 解析
- 为了减少重构风险，本次统一保留该格式

### 8.4 生成结果要求

生成结果必须写入结果编辑区，并满足：

- 必须是多轮对话
- 必须覆盖对话核心内容
- 尽量贴合使用场景
- 尽量不超过字数限制
- Speaker 数量与配置一致

### 8.5 生成成功后的行为

1. 将结果写入编辑区
2. 文本状态改为“已生成文本”
3. 生成文本文件
4. 启用“生成 mp3 音频”
5. 启用“下载文本”
6. 使旧音频失效
7. 若之前已有音频，则音频状态改为“音频已失效，请重新生成”

### 8.6 生成失败后的行为

1. 保留用户当前输入
2. 文本状态改为“文本生成失败”
3. 弹出错误提示
4. 不清空已有旧文本，除非本次是首次生成

### 8.7 再次生成文本

如果当前结果区已有文本，再点击“生成场景对话文本”，必须先弹确认框：

```text
重新生成将覆盖当前生成结果，是否继续？
```

确认后才允许覆盖。

## 9. 结果编辑区

### 9.1 作用

显示文本生成结果，并允许用户手动修改。

### 9.2 要求

1. 保持大文本框形式
2. 用户可直接编辑生成结果
3. 必须显式提示：

> 生成后的对话会显示在这里，可手动修改。生成 mp3 音频时，将使用当前编辑内容。

### 9.3 编辑后的联动

当文本框内容被修改后，必须立刻触发：

1. 文本状态改为“文本已修改”
2. 若已有音频，则音频状态改为“音频已失效，请重新生成”
3. 禁用“下载音频”
4. 保留“下载文本”和“生成 mp3 音频”可用

### 9.4 空文本规则

当编辑区为空或只包含空白时：

- “生成 mp3 音频”禁用
- “下载文本”禁用

## 10. 音频生成

### 10.1 按钮名称

`生成 mp3 音频`

### 10.2 目标

基于结果区当前文本生成 mp3 音频。

### 10.3 音频生成前置条件

1. 结果区文本非空
2. 当前未处于音频生成中
3. 文本状态不是无效状态
4. 文本长度未超过系统硬上限

### 10.4 文本长度硬上限

为保证本地 Demo 的稳定性，本次重构明确采用以下硬限制：

- 音频生成前，结果区文本最大长度为 `12000` 个字符

超过时：

- 前端禁止提交
- 提示用户缩短文本后重试
- 后端也需要做同样的兜底校验

### 10.5 行为要求

1. 点击后进入“音频生成中”
2. 默认输出格式固定为 `mp3`
3. 音频生成依据必须是**当前文本框内容**
4. 不重新请求文本生成
5. 成功后：
   - 音频状态改为“已生成音频”
   - 记录音频文件名
   - 启用“下载音频”
6. 失败后：
   - 音频状态改为“音频生成失败”
   - 保留文本内容
   - 不影响“下载文本”

### 10.6 音频生成中的编辑策略

为减少状态冲突，本次优化版 PRD明确采用以下策略：

- 音频生成中，结果编辑框临时只读
- 音频生成完成或失败后恢复可编辑

## 11. 下载文本

### 11.1 按钮名称

`下载文本`

### 11.2 前置条件

1. 当前有可下载文本
2. 结果区文本非空

### 11.3 唯一权威实现

本次重构统一采用以下实现，不再保留二选一口径：

1. 点击“下载文本”时，前端先把当前编辑区文本提交给后端保存
2. 后端保存成功后，下载该次保存生成的文本文件

也就是说：

- 不采用前端本地拼 txt 下载作为正式实现
- 页面展示的“当前文本文件”与实际下载的文本文件必须是同一个后端文件

### 11.4 文件规则

- 格式：`.txt`
- 内容：当前编辑区最新文本
- 文件名建议：`scene_dialogue_YYYYMMDD_HHMMSS.txt`

## 12. 下载音频

### 12.1 按钮名称

`下载音频`

### 12.2 前置条件

1. 当前已有已生成且未过期的音频
2. 当前音频状态为“已生成音频”

### 12.3 文件规则

- 格式：`.mp3`
- 文件名建议：`scene_dialogue_YYYYMMDD_HHMMSS.mp3`

### 12.4 关键规则

- 只要文本在音频生成后发生变化，旧音频立即失效
- 失效后“下载音频”按钮禁用，直到重新生成成功
- 本次不采用“下载音频时自动补生成”策略

## 13. 重置

### 13.1 按钮名称

`重置`

### 13.2 确认文案

```text
确定重置当前内容吗？已生成的文本和音频状态将被清空。
```

### 13.3 确认后的行为

1. 恢复人物画像区所有字段为默认值
2. 清空场景说明
3. 清空对话核心内容
4. 恢复人物数量默认值
5. 恢复字数限制默认值
6. 清空结果文本框
7. 清空文本状态
8. 清空音频状态
9. 清空当前文本文件信息
10. 清空当前音频文件信息
11. 禁用“生成 mp3 音频”
12. 禁用“下载文本”
13. 禁用“下载音频”

取消后不做任何处理。

## 14. 状态模型

### 14.1 文本状态

| 状态 | 含义 |
|---|---|
| `idle` | 未生成文本 |
| `generating` | 文本生成中 |
| `ready` | 已生成文本 |
| `edited` | 文本已修改 |
| `failed` | 文本生成失败 |

### 14.2 音频状态

| 状态 | 含义 |
|---|---|
| `idle` | 未生成音频 |
| `generating` | 音频生成中 |
| `ready` | 已生成音频 |
| `stale` | 音频已失效 |
| `failed` | 音频生成失败 |

### 14.3 当前文件展示

状态区保留四张卡片：

1. 文本状态
2. 音频状态
3. 当前文本文件
4. 当前音频文件

要求：

- 无文件时显示“未生成”
- 有文件时显示文件名和更新时间

|---|---:|---:|---:|---:|---:|
| 初始态 | 可用 | 禁用 | 禁用 | 禁用 | 可用 |
| 文本生成中 | 禁用 | 禁用 | 禁用 | 禁用 | 可用 |

| 文本已修改 | 可用 | 可用 | 可用 | 禁用 | 可用 |
| 音频生成中 | 视实现可禁用 | 禁用 | 可用 | 禁用 | 可用 |
| 已生成音频 | 可用 | 可用 | 可用 | 可用 | 可用 |
| 生成失败 | 可用 | 视文本状态决定 | 视文本状态决定 | 禁用 | 可用 |

## 20. 技术重构要求（给 Codex）

### 20.1 前端

1. 拆分页面状态管理，不把所有状态散落在按钮回调里
2. 所有按钮状态由统一状态派生
3. 文本状态和音频状态独立建模
4. 文本编辑区变更时统一触发音频失效联动
5. 文本生成中和音频生成中要有明确的 UI 锁定策略

### 20.2 后端

1. 保留本地可运行能力
2. 保留局域网访问能力
3. 文本生成与音频生成接口继续解耦
4. 统一文件输出目录管理
5. 后端不要把堆栈直接暴露给前端
6. 对文本格式和最大长度做兜底校验

### 20.3 文件管理

1. 文本与音频文件输出到同一次会话目录
2. 页面状态只展示“当前最新文件”
3. 文本下载与页面展示的“当前文本文件”必须一致

## 21. 验收标准

1. 页面仍是单页结构
2. 对话设置区字段与顺序保留
3. 按钮区按钮与顺序保留
